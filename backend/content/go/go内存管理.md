# go的内存模型以及堆的分配管理

go相比于c开发了一套自动的垃圾回收机制，所以编码的时候不用过多精通内存，但是必须要知道

说到内存管理，实际上是堆内存管理，栈内存是不需要程序去参与的

## TCMalloc

是Thread Cache Malloc的简称，是go内存管理起源

TCMalloc的做法：为每个线程预分配一块缓存，线程申请小内存的时候可以从缓存分配内存，这样有两个好处

*   为线程预分配缓存进行一次系统调用，后续线程申请小内存的时候直接从缓存分配，都是在用户态执行，没有系统调用，快速分配
*   多个线程同时申请小内存的时候，从各自缓存分配，无需加锁

## go的内存模型

go的内存分布布局分为以下几个区域：

*   代码段：编译后的指令，由操作系统管理
*   数据段：全局变量，静态变量，由编译器管理
*   栈区：每个goroutine独立栈，初始2-8kb，动态扩缩容，运行时自动管理
*   堆区：逃逸分析确定需要动态分配的对象，由gc+内存分配器管理

## go的堆分配管理

核心结论：go堆分配器 = 分级缓存（mcache / mcentral / mheap）+ 按大小分级 + 页式管理 + 并发友好设计

整体结构： goroutine -> mcache -> mcentral -> mheap -> OS

*   mcache（线程私有）：每个p有一个；小对象分配走这里；无锁，最快路径
*   mcentral（全局共享）：按size class分类管理span， mcache不够时从这里拿
*   mheap（全局堆）：管理所有span，不够时向OS申请内存

### 分配流程

#### 小对象（小于32kb）

1.  goroutine申请对象
2.  找到当前P的mcache
3.  根据size class找对应的span
4.  span里取一个空闲的slot
5.  返回指针

特点：无锁，O(1)，极快

#### 慢路径（mcache不够）

情况一：span用完：mcache->mcentral请求新span

情况二：mcentral也没有：mheap取span

情况三：mheap不够：OS申请

#### 大对象分配（>32kb）

直接走mheap

特点：开销大，容易触发gc
