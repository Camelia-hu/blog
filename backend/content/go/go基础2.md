# 有哪些情况必须手动对齐内存

内存对齐一般由编译器自动处理，但有几个特殊情况需要手动对齐：

1.  与c语言交互的场景，例如调用c库
2.  性能优化需求。例如处理大规模结构体数组时确保结构体对齐可以提供缓存命中率
3.  使用unsafe包直接操作内存
4.  特殊的二进制序列化/反序列化：如果需要将结构序列化为特定的二进制格式（如自定义的网络协议或存储文件），必须手动调节字段顺序和内存对齐
5.  与硬件交互

## 什么是内存对齐

是指数据在内存中存储位置满足一定的规则从而提高访问性能。通常，数据的地址应该是其类型大小的倍数

内存对齐的目的是让CPU可以高效的读取和写入数据

## 编译器的默认内存对齐策略

编译器会自动根据每个字段类型大小对齐结构体字段

主要的go对齐策略是会在结构体中插入适当的填充字节，确保字段对齐并提升访问性能

```go
type Example struct {
  A int8   // 占 1 字节
  B int64  // 占 8 字节
}

```

会在A后面插入7个字节的填充，以使B的地址是8的倍数

## 怎么手动调节内存对齐

调节字段顺序，将大字段放在前面

```go
type Optimized struct {
   B int64  // 占 8 字节
   A int8   // 占 1 字节
}

```

# 栈扩容和栈缩容

栈扩容和栈缩容是goroutine管理栈内存的动态调整机制，旨在高效利用内存并确保程序稳定运行

## 栈扩容

每个goroutine在创建时都会建立一个2kb的栈，当函数调用层级加深或者局部变量增多，导致栈空间不足，就会触发栈扩容，是通过runtime.morestack函数检测栈空间够不够，不够就创建新的栈，容量为两倍但不超过1GB。然后将旧栈内容复制到新栈

## 栈缩容

当栈内存利用率低于百分之25时就会触发缩容，少一半但不会低于2kb，缩容与扩容类似，都是用runtime.morestack检测并且开辟新的栈，将旧栈数据复制过去

# 分段栈

分段栈是go的栈管理机制的一种优化策略，但是现在已经不用了

## 为什么需要分段栈

传统语言每个线程启动时会分配一个固定大小的栈（1MB或2MB），但是大多数只用一小部分，浪费极大

go里面每个goroutine会分配一个会自动增长缩小的小栈（2kb），这使得go可以轻松创建上百万个goroutine

## 什么是分段栈

当栈空间不足时就会分配一个新栈（2kb或更大）将当前栈的数据全部迁移过去，将旧栈连接到新栈，形成链式结构的栈段

## 缺点

1.  性能开销大：每次函数调用都要检查剩余空间够不够，不时要触发迁移
2.  CPU分支预测失效
3.  垃圾回收复杂：多段链式结构不好统一扫描管理

## 连续栈

不够时会自动申请一个更大的连续内存块（通常为当前大小的两倍）

将原来栈的内容复制到新栈，更新栈指针

原栈给gc回收

# map的实现原理

map是基于哈希表实现的，使用桶数组存储键值对，map在存储时使用哈希表函数将键映射到相应的桶中，每个桶最多容纳8个键值对，超出后会通过一处同解决哈希冲突

当负载因子到达阈值时，map会触发扩容，并进行渐进式迁移，避免一次性扩容带来的性能损失

## map的底层结构

主要设计的结构有hmap和bmap

1.  hmap：整个map的元信息头
2.  bmap：桶，每个桶包含多个key-value
3.  overflow bucket：溢出桶，处理哈希冲突
4.  mapextra：额外信息，如溢出桶链表

### 核心结构体分析：

#### hmap：map的主结构

```go
type hmap struct {
    count     int            // 当前键值对个数
    flags     uint8          // 状态标志，比如是否在扩容中
    B         uint8          // bucket 数量是 2^B
    noverflow uint16         // 溢出桶估计数量
    hash0     uint32         // 哈希种子，防止 DoS 攻击

    buckets    unsafe.Pointer // 指向 buckets 数组（[]bmap）
    oldbuckets unsafe.Pointer // 扩容时的旧 buckets
    nevacuate  uintptr        // 当前已经搬迁的 bucket 下标

    extra *mapextra // overflow 数据、iterator 状态等
}
```
